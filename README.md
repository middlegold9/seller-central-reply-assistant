# seller-central-reply-assistant

> Seller Central 买家消息 AI 回复助手（Chrome 插件）—— 在卖家后台直接生成合规回复草稿。
> 最「当场击穿」的 Demo：客服是时差 + 量大的刚需，打开后台就有 AI 草稿，使用率不用愁（FDE 文档 9.3：嵌进工作流才有使用率）。

> 阶段：**功能设计（Design）**。本仓库当前为端到端功能梳理文档，代码骨架待落地。

---

## 0. 背景与动机

**跨境卖家客服痛点**
- **时差**：美国买家白天发消息 = 中国运营半夜；24h 内未回复伤 ODR / 评分。
- **量大**：促销季消息暴涨，手工逐条写。
- **政策红线**：亚马逊严禁用消息索评、引导站外、操纵排名；违规会封号。
- **多语言**：买家用英 / 西 / 德 / 日等，卖家需对应语言回复。
- **话术参差**：新人易写错 / 踩线。

插件把 AI 直接注射进 Seller Central 的 Buyer Messages 页面，一键生成「合规、贴合品牌、带政策风险提示」的回复草稿，人工确认后发送。

---

## 1. 目标与范围

**Goals**
- 在 Buyer Messages 页注入「AI 草稿」入口，零切换。
- 自动抓当前会话上下文（买家消息、订单号、商品）。
- 生成合规草稿 + 政策风险标记 + 建议动作（退款 / 换货 / RMA）。
- 高风险（法律 / 威胁 / A-to-z / 差评移除请求）自动转人工。
- 多语言生成；品牌语气可配置。

**Personas**：跨境卖家客服、代运营（多店铺）。

**Non-goals**
- 不自动发送（必须人工确认）。
- 不替代亚马逊消息系统（仍走官方发送通道，避免违规）。
- v1 不做自动退款执行。

## 2. 端到端链路（核心：把每一步写清）

```
[卖家打开 Seller Central → 买家消息页]
   │ 1. Content Script 注入
   ▼
[扩展识别当前会话]
   │ 2. 解析 DOM：买家最新消息、订单ID、商品ASIN、消息类型
   ▼
[点击「AI 草稿」按钮]
   │ 3. 收集上下文（消息文本 + 订单ID + 商品ASIN + 市场 + 语言）
   ▼
[扩展 popup/background → 后端 API]
   │ 4. 后端可选调用 sp-api-mcp-server：
   │      spapi_orders_get(状态/物流) + spapi_orders_buyer_info(RDT, 姓名/语言偏好)
   ▼
[政策护栏引擎] ← 规则库（见 §3.3）
   │ 5. 先判定消息分类 + 风险等级
   ▼
[LLM 生成草稿]（system prompt 内嵌政策红线 + 品牌语气 + 场景模板）
   │ 6. 生成多语言草稿 + 建议动作
   ▼
[政策护栏二次校验]
   │ 7. 扫描草稿是否含违禁词/违规意图 → 标红 + 阻断高危
   ▼
[扩展渲染：草稿框 + 风险标签 + 动作按钮]
   │ 8. 卖家编辑
   ▼
[卖家点「插入并发送」]
   │ 9. 草稿填回消息输入框（走官方发送，不绕过）
   ▼
[发送后]
   │ 10. 记录（脱敏）到学习库 + 埋点（耗时/是否采用/是否转人工）
```

## 3. 功能模块

### 3.1 上下文抓取（Content Script）
- 监听 Buyer Messages 页路由，识别会话容器。
- 解析：买家最近消息、订单号（从链接 / 文本提取）、商品 ASIN、消息时间、买家语言（检测）。
- 容错：DOM 结构变化 → 选择器可配置 + 兜底手动粘贴。

### 3.2 AI 草稿生成
- 后端 LLM（可配置：OpenAI / Claude / 本地）。
- Prompt 结构：系统（政策红线 + 品牌语气 + 场景模板）+ 用户（上下文 + 买家消息）。
- 输出结构化 JSON：`{draft, language, suggested_actions[], risk_flags[], confidence}`。

### 3.3 政策护栏引擎（核心卖点）
规则库（可更新），分级：
- **致命（阻断发送）**：索评 / 求好评（"please leave a review"）、引导站外（邮箱 / 微信 / URL）、承诺移除差评、虚假物流、歧视 / 攻击。
- **警告（标红提醒）**：过度承诺时效、未授权折扣、要求改评价内容。
- **提示**：建议用官方模板、建议先查物流再回复。
- 引擎对「买家消息」和「AI 草稿」各跑一遍（输入侧防误判、输出侧防违规）。
- 规则来源：亚马逊 Buyer-Seller Messaging 政策 + 社区踩坑库，可热更新。

### 3.4 消息分类
- `logistics`（物流 / 时效）、`quality`（质量 / 破损）、`refund`（退款）、`feedback_removal`（差评移除请求）、`a_to_z`（索赔）、`other`。
- 分类驱动模板与动作建议。

### 3.5 多语言
- 检测买家语言 → 用同语言回复；支持英 / 西 / 德 / 法 / 日 / 意等。
- 内置术语表（"tracking number" 等）避免机翻错误。

### 3.6 建议动作
- 基于订单状态 + 政策给可执行建议：部分退款金额、换货、重发、RMA 指引、升级。
- 退款额度可设上限（防止 AI 瞎给大额）。

### 3.7 高风险转人工
- 命中「法律 / 威胁 / A-to-z / 差评移除」→ 不生成发送草稿，改为「建议人工处理」卡片 + 要点摘要。

### 3.8 历史学习（RAG）
- 把历史「高采用率 + 低风险」回复建索引；新会话检索相似案例，提升话术质量。
- 仅存脱敏文本（去 PII）。

### 3.9 多店铺 & 设置
- 店铺切换（不同品牌语气 / 模板 / 语言）。
- Options 页：API Key（用户自填，存 chrome.storage 本地）、品牌名 / 签名、语气、退款上限、护栏严格度。

---

## 4. 架构
- 前端：Plasmo（Chrome MV3）。Popup + Content Script + Background Service Worker + Options。
- 后端（可自托管 / 可选）：FastAPI
  - `/draft`：生成草稿（调 LLM + 护栏）
  - `/classify`：分类
  - `/enrich`：调 sp-api-mcp 丰富上下文（可选）
  - `/learn`：写入学习库
- 存储：chrome.storage.local（用户设置 / Key）；后端 Postgres / DuckDB（学习库、埋点）。
- 通信：扩展 ↔ 后端 HTTPS；后端 ↔ MCP（stdio / SSE）。

---

## 5. 配置与部署
- `.env.example`：`LLM_BASE_URL`, `LLM_KEY`, `MCP_BASE_URL`, `GUARDRAIL_STRICT`。
- 构建：`pnpm build`（Plasmo）→ `chrome://extensions` 加载 `build` 或上架 Web Store。
- 后端：`docker-compose up`。

---

## 6. 安全与合规（重中之重）
- **亚马逊政策**：绝不索评、绝不引导站外、不操纵排名；护栏硬阻断。
- **数据最小化**：仅取生成回复所需上下文；PII 不持久化（或脱敏）。
- **本地优先**：API Key 存本地，不出用户设备；后端可选关闭。
- **发送走官方通道**：插件只把草稿填回输入框，不模拟提交，避免违反 API ToS。
- **审计**：所有生成记录留痕（时间 / 订单 / 风险等级），便于复盘。

---

## 7. 可验证
- **政策合规率**：构造含违规意图的测试集（如买家问「能给我折扣换好评吗」），断言护栏阻断 / 标红。目标 ≥ 99% 通过。
- **草稿可用率**：内部盲测，资深客服对草稿「可直接发 / 小改 / 重写」评分。
- **响应时延**：< 3s（p95）。
- **多语言正确率**：母语者抽检。
- **人工接管率**：上线后监控（越低越好，但高危必须接管）。

---

## 8. 可复盘
- 每周复盘：被转人工的会话、被标红的草稿 → 提炼新规则进护栏库。
- 采用率 / 编辑度埋点 → 反哺 prompt 与模板。
- 差评 / 退款根因归类 → 回流产品 / 供应链（可复盘闭环）。

---

## 9. 与另两个项目的关系
- 依赖 **sp-api-mcp-server** 的 `spapi_orders_*` 丰富上下文（可选增强）。
- 与 **amazon-ads-review-agent** 无直接耦合，但共享「政策护栏」原则。

---

## 10. 路线图
- **M1**：Content Script 注入 + 单店铺 + 直连 LLM + 基础护栏。
- **M2**：政策护栏引擎 + 多语言 + 分类 + 风险转人工。
- **M3**：接 sp-api-mcp 丰富上下文 + RAG 学习库。
- **M4**：多店铺 + Web Store 上架 + 埋点面板。

---

## 11. 风险与开放问题
- Seller Central DOM 常变 → 选择器需可配置 + 探测兜底。
- 亚马逊可能限制自动化插件 → 仅「辅助生成、人工发送」降低风险。
- LLM 幻觉给错物流信息 → 强制以 sp-api 实数据为准，不编。
